# TCP/IP Advanced

# Chapter 05

# Sliding Window 심화

------------------------------------------------------------------------

# 학습 목표

이번 장에서는 TCP의 핵심 기술인 **Sliding Window**를 학습한다.

학습 후에는 다음 내용을 설명할 수 있어야 한다.

-   Sliding Window란 무엇인가?
-   Sliding Window가 필요한 이유
-   송신 카테고리와 수신 카테고리
-   Window가 이동하는 원리
-   ACK와 Sliding Window의 관계
-   Buffer와 Window의 관계

------------------------------------------------------------------------

# 1. Sliding Window란?

TCP는 데이터를 하나 전송한 뒤 ACK를 기다리는 방식(Stop-and-Wait)으로
동작하지 않는다.

대신 **ACK를 기다리지 않고 Window 크기만큼 여러 데이터를 연속으로
전송**할 수 있다.

이를 **Sliding Window**라고 한다.

``` text
Data1  Data2  Data3  Data4
  │      │      │      │
  └──────┴──────┴──────┘
      ACK 없이 연속 전송
```

------------------------------------------------------------------------

# 2. 왜 Sliding Window가 필요한가?

Stop-and-Wait 방식은 RTT가 길수록 성능이 크게 떨어진다.

예를 들어 1000Byte를 전송한다고 하면

``` text
100Byte 전송
   │
   ▼
 ACK 수신
   │
   ▼
100Byte 전송
```

를 반복해야 한다.

반면 Window가 500Byte라면

``` text
500Byte 전송
500Byte 전송
      │
      ▼
    ACK 수신
```

처럼 네트워크를 훨씬 효율적으로 사용할 수 있다.

------------------------------------------------------------------------

# 3. Window란?

Window는 **ACK를 받기 전에 송신자가 연속해서 보낼 수 있는 최대
데이터(Byte)** 를 의미한다.

이 값은 수신자가 광고(Advertise)하는 **Receive Window**를 기반으로
결정된다.

Window가 클수록 더 많은 데이터를 연속 전송할 수 있다.

------------------------------------------------------------------------

# 4. 송신 카테고리

교수님 PDF에서는 송신 데이터를 네 가지 영역으로 구분한다.

  카테고리   의미
  ---------- ---------------------------------------------
  1          전송했고 ACK까지 받은 데이터
  2          전송했지만 ACK를 받지 못한 데이터
  3          전송 가능한 데이터
  4          아직 Window 밖에 있어 전송할 수 없는 데이터

``` text
|----1----|----2----|----3----|----4----|
 ACK완료     ACK대기    전송가능    전송불가
```

Window는 **카테고리 2와 카테고리 3 영역**으로 구성된다.

------------------------------------------------------------------------

# 5. 수신 카테고리

수신 측도 비슷한 상태를 관리한다.

  카테고리   의미
  ---------- --------------------------
  1          정상적으로 수신한 데이터
  2          ACK를 전송한 데이터
  3          다음에 받을 데이터
  4          아직 받을 수 없는 데이터

------------------------------------------------------------------------

# 6. Window는 왜 이동하는가?

ACK가 도착하면

-   승인된 데이터가 Window에서 제거되고
-   새로운 전송 공간이 생긴다.

그 결과 Window가 앞으로 이동한다.

``` text
전송 전

[□□□□□□□□]

      │ ACK

      ▼

[    □□□□□□□□]
```

Window가 앞으로 **미끄러지듯(Slide)** 이동하기 때문에 Sliding Window라고
한다.

------------------------------------------------------------------------

# 7. ACK와 Sliding Window

ACK는 단순히

> "잘 받았습니다."

라는 의미만 있는 것이 아니다.

동시에

> "이제 다음 데이터를 보내도 됩니다."

라는 의미도 가진다.

ACK Number가 증가하면 송신 Window도 앞으로 이동한다.

------------------------------------------------------------------------

# 8. Window와 Buffer

Window는 수신자의 **남아 있는 Receive Buffer 크기**를 광고한 값이다.

예를 들어

``` text
Receive Buffer : 1000Byte
Advertised Window : 1000Byte
```

이면 송신자는 최대 1000Byte까지 ACK 없이 보낼 수 있다.

Buffer가 가득 차면 Window 값은 감소하고,

Buffer가 비워지면 Window는 다시 증가한다.

------------------------------------------------------------------------

# 9. 교수님 수업 포인트

-   Window는 Buffer와 밀접한 관계가 있다.
-   ACK가 도착해야 Window가 이동한다.
-   Sliding Window는 TCP Flow Control의 핵심이다.
-   Sequence Number와 ACK Number를 함께 이해해야 한다.

------------------------------------------------------------------------

# 10. 실무에서는?

Wireshark에서는 다음 항목을 함께 확인한다.

-   Window Size
-   Sequence Number
-   ACK Number

세 값을 보면 Window가 정상적으로 이동하는지 분석할 수 있다.

------------------------------------------------------------------------

# 시험 포인트

-   Sliding Window는 ACK를 기다리지 않고 여러 데이터를 전송하기 위한
    기술이다.
-   Window는 수신자가 광고한 Receive Buffer 크기를 기반으로 한다.
-   ACK가 도착하면 Window가 앞으로 이동한다.
-   Sliding Window는 TCP Flow Control의 핵심 기술이다.

------------------------------------------------------------------------

# 핵심 요약

-   Sliding Window는 TCP의 전송 효율을 높이는 핵심 기술이다.
-   ACK를 기다리지 않고 여러 데이터를 연속 전송할 수 있다.
-   Window는 수신자의 Receive Buffer를 기반으로 결정된다.
-   ACK가 증가하면 Window도 함께 앞으로 이동한다.
-   Sequence Number와 ACK Number를 이용하여 데이터 전송을 제어한다.
