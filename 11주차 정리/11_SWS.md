# Silly Window Syndrome (SWS)

------------------------------------------------------------------------

# 학습 목표

이번 장에서는 TCP에서 발생할 수 있는 성능 저하 현상인 **Silly Window Syndrome(SWS)** 를 학습한다.

학습 후에는 다음 내용을 설명할 수 있어야 한다.

- SWS란 무엇인가?
- 왜 발생하는가?
- 어떤 문제가 생기는가?
- 송신 측 해결 방법
- 수신 측 해결 방법

------------------------------------------------------------------------

# 1. SWS란?

SWS(Silly Window Syndrome)는 **매우 작은 크기의 데이터를 반복적으로 전송하여 네트워크 효율이 크게 떨어지는 현상**이다.

TCP는 신뢰성 있는 통신을 제공하지만, 너무 작은 데이터를 계속 전송하면 TCP Header의 비중이 커져 전송 효율이 크게 저하된다.

> 보낼 수 있다고 해서 너무 작은 데이터를 계속 보내는 현상이다.

------------------------------------------------------------------------

# 2. 왜 문제가 되는가?

TCP Header는 기본적으로 20Byte이다.

예를 들어

```text
TCP Header : 20Byte
Payload    : 1Byte
```

라면

```text
총 전송 크기 : 21Byte

실제 데이터 : 1Byte
```

뿐이다.

즉,

```text
Header : 20Byte
Data   : 1Byte
```

가 되어 Header 오버헤드가 매우 커지고 네트워크 효율이 크게 떨어진다.

------------------------------------------------------------------------

# 3. 발생 원인

대표적인 원인은 다음과 같다.

- 작은 Window
- 작은 Payload
- 잦은 ACK
- 애플리케이션이 Receive Buffer를 조금씩만 소비

SWS는 **송신 측(Sender)** 과 **수신 측(Receiver)** 모두에서 발생할 수 있다.

------------------------------------------------------------------------

# 4. 송신 측(Sender)에서 발생하는 경우

송신 가능한 Window가 10Byte라고 가정하자.

```text
10Byte 전송
   │
   ▼
 ACK 수신
   │
   ▼
10Byte 전송
   │
   ▼
 ACK 수신
```

작은 Segment가 계속 생성되면서

- Header 증가
- 패킷 증가
- 효율 저하

가 발생한다.

------------------------------------------------------------------------

# 5. 수신 측(Receiver)에서 발생하는 경우

Receive Buffer가

```text
1000Byte
```

인데,

애플리케이션이

```text
5Byte
```

씩만 읽는다면

수신자는 계속

```text
Window = 5
```

를 광고하게 된다.

송신자는

```text
5Byte
```

씩만 계속 보내게 되어

SWS가 발생한다.

------------------------------------------------------------------------

# 6. 문제점

SWS가 발생하면

- 패킷 수 증가
- Header 오버헤드 증가
- CPU 사용량 증가
- 네트워크 대역폭 낭비
- 전체 처리량(Throughput) 감소

가 발생한다.

------------------------------------------------------------------------

# 7. 해결 방법 ① Clark Solution

Clark Solution은 **수신 측(Receiver)** 해결 방법이다.

버퍼가 조금 비었다고 바로 작은 Window를 광고하지 않고,

충분한 공간이 확보될 때까지 기다렸다가

한 번에 큰 Window를 광고한다.

예를 들어

```text
Window = 5

↓

광고하지 않음

↓

Window = 500

↓

광고
```

이렇게 작은 Window 광고를 방지하여

SWS를 예방한다.

------------------------------------------------------------------------

# 8. 해결 방법 ② Nagle Algorithm

Nagle Algorithm은 **송신 측(Sender)** 해결 방법이다.

작은 데이터를 즉시 보내지 않고,

이전에 전송한 데이터의 ACK가 도착할 때까지 기다리거나,

충분한 데이터가 모이면 한 번에 전송한다.

예를 들어

```text
1B
2B
3B
4B

↓

10B 전송
```

처럼 여러 작은 데이터를 하나의 Segment로 묶어 전송한다.

> 지연에 민감한 응용(예: 온라인 게임, VoIP, 원격 터미널(SSH 등))에서는
> Nagle Algorithm을 비활성화(TCP_NODELAY)하기도 한다.
>
> 특히 대부분의 SSH 구현(OpenSSH 등)은 키 입력의 응답성을 높이기 위해
> TCP_NODELAY를 사용하여 Nagle Algorithm을 비활성화한다.

------------------------------------------------------------------------

# 9. Clark Solution과 Nagle Algorithm 비교

| 구분 | Clark Solution | Nagle Algorithm |
|------|----------------|-----------------|
| 적용 위치 | 수신 측 | 송신 측 |
| 목적 | 작은 Window 광고 방지 | 작은 Payload 전송 방지 |
| 동작 방식 | 충분한 공간 확보 후 Window 광고 | 데이터를 모아서 전송 |
| 효과 | 작은 Window 감소 | 작은 Segment 감소 |

------------------------------------------------------------------------

# 10. 교수님 수업 포인트

- SWS는 작은 데이터를 반복 전송하여 발생한다.
- Header 오버헤드가 증가한다.
- Clark Solution은 수신 측 해결 방법이다.
- Nagle Algorithm은 송신 측 해결 방법이다.
- 두 기법은 함께 사용되어 TCP 효율을 높인다.

------------------------------------------------------------------------

# 11. Wireshark에서는?

SWS가 발생하면

작은 Payload를 가진 TCP Segment가 연속해서 관찰된다.

예를 들어

```text
Length = 1
Length = 2
Length = 3
Length = 1
Length = 2
```

처럼 매우 작은 Segment가 반복적으로 전송되는 모습을 볼 수 있다.

------------------------------------------------------------------------

# 12. 실무 예시

Telnet처럼 작은 키 입력이 자주 발생하는 응용에서는

Nagle Algorithm을 사용하여 여러 입력을 묶어 전송하면

네트워크 효율을 높일 수 있다.

반면 SSH(OpenSSH 등)는 사용자의 키 입력에 대한

즉각적인 응답이 중요하므로

대부분 TCP_NODELAY를 사용하여

Nagle Algorithm을 비활성화한다.

온라인 게임과 VoIP 역시

지연 시간을 최소화하기 위해

Nagle Algorithm을 비활성화하는 경우가 많다.

# 핵심 요약

- SWS는 작은 Payload를 반복 전송하여 발생하는 TCP 성능 저하 현상이다.
- Header 오버헤드가 증가하여 네트워크 효율이 떨어진다.
- 송신 측과 수신 측 모두에서 발생할 수 있다.
- Clark Solution은 작은 Window 광고를 방지하는 수신 측 해결 방법이다.
- Nagle Algorithm은 작은 데이터를 모아 전송하는 송신 측 해결 방법이다.
- SSH, 온라인 게임, VoIP 등 지연에 민감한 응용에서는 TCP_NODELAY를 사용하여 Nagle Algorithm을 비활성화하는 경우가 많다.
