# TCP/IP Advanced

# Chapter 11

# Silly Window Syndrome (SWS)

------------------------------------------------------------------------

# 학습 목표

이번 장에서는 TCP에서 발생할 수 있는 성능 저하 현상인 **Silly Window
Syndrome(SWS)** 를 학습한다.

학습 후에는 다음 내용을 설명할 수 있어야 한다.

-   SWS란 무엇인가?
-   왜 발생하는가?
-   어떤 문제가 생기는가?
-   송신 측 해결 방법
-   수신 측 해결 방법

------------------------------------------------------------------------

# 1. SWS란?

SWS(Silly Window Syndrome)는 **매우 작은 크기의 데이터를 반복적으로
전송하여 네트워크 효율이 크게 떨어지는 현상**이다.

> 보낼 수 있다고 해서 너무 작은 데이터를 계속 보내는 현상이다.

------------------------------------------------------------------------

# 2. 왜 문제가 되는가?

TCP Header는 기본적으로 20Byte이다.

예를 들어

``` text
TCP Header : 20Byte
Payload : 1Byte
```

라면 총 21Byte를 전송하지만 실제 데이터는 1Byte뿐이다.

Header 오버헤드가 매우 커져 전송 효율이 떨어진다.

------------------------------------------------------------------------

# 3. 발생 원인

대표적인 원인은 다음과 같다.

-   작은 Window
-   작은 Payload
-   잦은 ACK
-   애플리케이션이 Receive Buffer를 조금씩만 소비

SWS는 송신 측과 수신 측 모두에서 발생할 수 있다.

------------------------------------------------------------------------

# 4. 송신 측에서 발생하는 경우

Window가 10Byte라고 가정하면

``` text
10Byte 전송
   │
   ▼
ACK
   │
   ▼
10Byte 전송
```

을 반복하게 되어 작은 Segment가 계속 생성된다.

------------------------------------------------------------------------

# 5. 수신 측에서 발생하는 경우

Receive Buffer가 1000Byte인데 애플리케이션이 매번 5Byte씩만 읽는다면

``` text
Window = 5
```

를 계속 광고하게 되고 송신자는 5Byte씩만 계속 전송하게 된다.

------------------------------------------------------------------------

# 6. 문제점

-   패킷 수 증가
-   Header 오버헤드 증가
-   CPU 사용량 증가
-   네트워크 대역폭 낭비
-   전체 처리량 감소

------------------------------------------------------------------------

# 7. 해결 방법 ① Clark Solution

Clark Solution은 **수신 측** 해결 방법이다.

버퍼가 조금 비었다고 바로 작은 Window를 광고하지 않고 충분한 공간이
확보될 때까지 기다린다.

``` text
Window = 5

(광고 안 함)

↓

Window = 500

↓

광고
```

------------------------------------------------------------------------

# 8. 해결 방법 ② Nagle Algorithm

Nagle Algorithm은 **송신 측** 해결 방법이다.

작은 데이터를 즉시 보내지 않고 ACK를 기다리거나 데이터를 모아서 한 번에
전송한다.

``` text
1B + 2B + 3B + 4B

↓

10B 전송
```

> 지연에 민감한 응용(예: 일부 게임, VoIP, SSH의 특정 상황)에서는 Nagle
> Algorithm을 비활성화하기도 한다.

------------------------------------------------------------------------

# 9. Clark Solution과 Nagle Algorithm 비교

  구분        Clark Solution               Nagle Algorithm
  ----------- ---------------------------- ------------------------
  적용 위치   수신 측                      송신 측
  목적        작은 Window 광고 방지        작은 Payload 전송 방지
  효과        충분한 공간 후 Window 광고   데이터를 모아서 전송

------------------------------------------------------------------------

# 10. 교수님 수업 포인트

-   SWS는 작은 데이터를 계속 보내 발생한다.
-   Clark Solution은 수신 측 해결 방법이다.
-   Nagle Algorithm은 송신 측 해결 방법이다.

------------------------------------------------------------------------

# 11. Wireshark에서는?

SWS가 발생하면 작은 Payload를 가진 TCP Segment가 연속해서 관찰된다.

``` text
Length = 1
Length = 2
Length = 3
Length = 1
```

------------------------------------------------------------------------

# 12. 실무 예시

Telnet이나 SSH처럼 키 입력이 자주 발생하는 환경에서는 작은 패킷이 계속
생성될 수 있다.

운영체제는 Nagle Algorithm을 적용하여 여러 입력을 묶어 전송함으로써
효율을 높일 수 있다.

------------------------------------------------------------------------

# 시험 포인트

-   SWS는 작은 데이터를 반복 전송하여 성능이 저하되는 현상이다.
-   Header 오버헤드가 증가한다.
-   Clark Solution은 수신 측 해결 방법이다.
-   Nagle Algorithm은 송신 측 해결 방법이다.

------------------------------------------------------------------------

# 암기법

``` text
SWS → 작은 데이터

Clark → 수신 측

Nagle → 송신 측
```

# 핵심 요약

-   SWS는 작은 Payload를 반복 전송하여 발생하는 성능 저하 현상이다.
-   작은 Window와 작은 Payload가 주요 원인이다.
-   Clark Solution은 작은 Window 광고를 방지한다.
-   Nagle Algorithm은 작은 데이터를 모아 전송한다.
-   두 기법 모두 네트워크 효율을 향상시키기 위해 사용된다.
