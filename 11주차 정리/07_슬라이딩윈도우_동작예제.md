# Chapter 07 — Sliding Window 동작 예제

## 학습 목표

- 3-Way Handshake 직후 초기 포인터 상태 파악
- 데이터 송수신 과정에서 SND / RCV 포인터 변화 추적
- Window 크기가 변경되는 시점 이해
- Wireshark 패킷으로 포인터 추론

## 1. 초기 상태 (3-Way Handshake 완료 직후)

| 항목 | Client | Server |
|:---|:---:|:---:|
| ISN | 1 | 241 |
| Receive Window | 360 Byte | 200 Byte |

### Client 포인터

```
SND.UNA = 1     ← 아직 보낸 데이터 없음
SND.NXT = 1     ← SND.UNA와 같음 (미전송)
SND.WND = 360   ← Server가 광고한 Receive Window

RCV.NXT = 241   ← Server ISN 기준 다음 수신 번호
RCV.WND = 200   ← Client Receive Buffer 여유
```

### Server 포인터

```
SND.UNA = 241   ← 아직 보낸 데이터 없음
SND.NXT = 241   ← SND.UNA와 같음 (미전송)
SND.WND = 200   ← Client가 광고한 Receive Window

RCV.NXT = 1     ← Client ISN 기준 다음 수신 번호
RCV.WND = 360   ← Server Receive Buffer 여유
```

> SND.UNA = SND.NXT 이면 **전송한 데이터가 없거나 모두 ACK 받은 상태**다.

## 2. 전체 동작 흐름 (10단계)

```
Client                          Server
  │                               │
  │──── ① 140 Byte 요청 ────────▶│
  │                               │
  │◀─── ② ACK141 + 80 Byte ──────│
  │                               │
  │──── ③ ACK321 ───────────────▶│
  │                               │
  │◀─── ④ Part1 (120 Byte) ──────│
  │                               │
  │──── ⑤ ACK441 ───────────────▶│
  │                               │
  │◀─── ⑥ Part2 (80 Byte) ───────│
  │                               │
  │──── ⑦ ACK521 ───────────────▶│
  │                               │
  │◀─── ⑧ Part3 (80 Byte) ───────│
  │                               │
  │──── ⑨ ACK601 ───────────────▶│
```

## 3. 단계별 포인터 변화

### Client 포인터 변화

| 단계 | 동작 | SND.UNA | SND.NXT | SND.WND | RCV.NXT | RCV.WND |
|:---:|:---|:---:|:---:|:---:|:---:|:---:|
| 1 | 연결 수립 | 1 | 1 | 360 | 241 | 200 |
| 2 | 140 Byte 요청 전송 | 1 | **141** | 360 | 241 | 200 |
| 3 | ACK141 + 80 Byte 응답 수신 | **141** | 141 | **260** | **321** | 200 |
| 4 | ACK321 전송 | 141 | 141 | 260 | 321 | 200 |
| 5 | Part1 120 Byte 수신 | 141 | 141 | 260 | **441** | 200 |
| 6 | ACK441 전송 | 141 | 141 | 260 | 441 | 200 |
| 7 | Part2 80 Byte 수신 | 141 | 141 | 260 | **521** | 200 |
| 8 | ACK521 전송 | 141 | 141 | 260 | 521 | 200 |
| 9 | Part3 80 Byte 수신 | 141 | 141 | 260 | **601** | 200 |
| 10 | ACK601 전송 | 141 | 141 | 260 | 601 | 200 |

> **SND.WND가 360 → 260으로 바뀐 이유:**  
> 3단계에서 Server가 ACK141과 함께 자신의 Receive Window(260 Byte)를 광고했기 때문이다.  
> Window 크기는 **데이터 전송 시가 아니라, 상대방의 ACK를 수신할 때 갱신된다.**

---

### Server 포인터 변화

| 단계 | 동작 | SND.UNA | SND.NXT | SND.WND | RCV.NXT | RCV.WND |
|:---:|:---|:---:|:---:|:---:|:---:|:---:|
| 1 | 연결 수립 | 241 | 241 | 200 | 1 | 360 |
| 2 | 140 Byte 요청 수신 | 241 | 241 | 200 | **141** | **260** |
| 3 | ACK141 + 80 Byte 전송 | 241 | **321** | 200 | 141 | 260 |
| 4 | ACK321 수신 | **321** | 321 | 200 | 141 | **360** |
| 5 | Part1 120 Byte 전송 | 321 | **441** | 200 | 141 | 360 |
| 6 | ACK441 수신 | **441** | 441 | 200 | 141 | 360 |
| 7 | Part2 80 Byte 전송 | 441 | **521** | 200 | 141 | 360 |
| 8 | ACK521 수신 | **521** | 521 | 200 | 141 | 360 |
| 9 | Part3 80 Byte 전송 | 521 | **601** | 200 | 141 | 360 |
| 10 | ACK601 수신 | **601** | 601 | 200 | 141 | 360 |

> **Server RCV.WND가 360 → 260으로 줄어든 이유:**  
> 2단계에서 Client의 140 Byte 요청이 Server Receive Buffer에 저장되었기 때문이다.  
> 360 − 140 = **260 Byte** 남음.  
> 4단계에서 ACK321(Client 응답)을 받으면서 Buffer가 비워져 다시 360으로 회복된다.

## 4. Sliding Window 이동 원리 정리

```
① 140 Byte 전송
        │
        ▼
② ACK141 수신
        │
        ▼
③ SND.UNA : 1 → 141 이동
        │
        ▼
④ Sliding Window 앞으로 이동
        │
        ▼
⑤ 다음 데이터 전송 가능
```

> Window는 데이터를 전송할 때 이동하는 것이 아니라  
> **ACK를 수신하여 SND.UNA가 앞으로 이동하는 순간 함께 이동한다.**

---

## 5. Window 범위를 초과하면?

Client가 Server로부터 **Receive Window = 260 Byte** 를 광고받았다.

만약 보내야 할 데이터가 280 Byte라면?

```
현재 Window = 260 Byte
보내야 할 데이터 = 280 Byte
→ 한 번에 전송 불가

260 Byte 전송
    │
    ▼
ACK 수신 → Window 이동
    │
    ▼
나머지 20 Byte 전송
```

TCP는 항상 **현재 허용된 Window 범위 내에서만** 데이터를 전송한다.

## 6. Wireshark에서 포인터 추론

| Wireshark 필드 | 추론 가능한 포인터 |
|:---|:---:|
| Sequence Number | SND.NXT |
| ACK Number | SND.UNA, RCV.NXT |
| Window Size | RCV.WND |

### 예시 패킷

```
Packet 1  Seq=1   Ack=241  Win=360   ← Client → Server (요청 전)
Packet 2  Seq=241 Ack=141  Win=260   ← Server → Client (ACK + 응답)
Packet 3  Seq=141 Ack=321  Win=200   ← Client → Server (ACK)
Packet 4  Seq=321 Ack=141  Win=360   ← Server → Client (Part1)
```

---

## 시험 포인트

- SND.NXT는 **데이터 전송 시** 증가한다
- SND.UNA는 **ACK 수신 시** 이동한다
- SND.WND는 **상대방 ACK에 포함된 Window 광고 수신 시** 갱신된다
- Window 범위를 **초과하여 전송할 수 없다**
- Sliding Window는 **ACK를 기준**으로 이동한다

## 핵심 요약

| 포인터 | 변화 시점 | 방향 |
|:---:|:---|:---|
| SND.NXT | 데이터 전송 시 | 증가 |
| SND.UNA | ACK 수신 시 | 이동 (앞으로) |
| SND.WND | ACK 수신 시 (Window 광고 포함) | 갱신 |
| RCV.NXT | 데이터 수신 시 | 증가 |
| RCV.WND | 버퍼 사용량 변화 시 | 증감 |
