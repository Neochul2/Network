# 1. 초기 상태

TCP 3-Way Handshake가 완료된 직후의 상태이다.

|항목|Client|Server|
|---|---:|---:|
|ISN|1|241|
|Receive Window|360Byte|200Byte|

## Client

```text
SND.UNA = 1
SND.NXT = 1
SND.WND = 360

RCV.NXT = 241
RCV.WND = 200
```

## Server

```text
SND.UNA = 241
SND.NXT = 241
SND.WND = 200

RCV.NXT = 1
RCV.WND = 360
```

> 아직 데이터를 보내지 않았으므로 SND.UNA와 SND.NXT는 같은 값을 가진다.

---

# 2. 전체 동작 흐름 (10단계)

1. 연결 수립
2. Client → 140Byte 요청 전송
3. Server → ACK141 + 80Byte 응답
4. Client → ACK321
5. Server → 파일 Part1(120Byte)
6. Client → ACK441
7. Server → 파일 Part2(80Byte)
8. Client → ACK521
9. Server → 파일 Part3(80Byte)
10. Client → ACK601

---

# 3. 포인터 변화 (교수님 PDF 기준)

## Client

|단계|동작|SND.UNA|SND.NXT|SND.WND|RCV.NXT|RCV.WND|
|---|---|---:|---:|---:|---:|---:|
|1|연결 수립|1|1|360|241|200|
|2|140Byte 요청 전송|1|141|360|241|200|
|3|ACK141 + 80Byte 응답 수신|141|141|260|321|200|
|4|ACK321 전송|141|141|260|321|200|
|5|120Byte(Part1) 수신|141|141|260|441|200|
|6|ACK441 전송|141|141|260|441|200|
|7|80Byte(Part2) 수신|141|141|260|521|200|
|8|ACK521 전송|141|141|260|521|200|
|9|80Byte(Part3) 수신|141|141|260|601|200|
|10|ACK601 전송|141|141|260|601|200|

> **Client의 SND.WND는 Server가 ACK141과 함께 광고한 Receive Window(260Byte)를 수신한 3단계에서 360Byte → 260Byte로 변경된다. Window 크기는 ACK에 포함된 Window Advertisement를 수신할 때 갱신된다.**

---

## Server

|단계|동작|SND.UNA|SND.NXT|SND.WND|RCV.NXT|RCV.WND|
|---|---|---:|---:|---:|---:|---:|
|1|연결 수립|241|241|200|1|360|
|2|140Byte 요청 수신 완료|241|241|200|141|260|
|3|ACK141 + 80Byte 응답 전송|241|321|200|141|260|
|4|ACK321 수신|321|321|200|141|360|
|5|120Byte(Part1) 전송|321|441|200|141|360|
|6|ACK441 수신|441|441|200|141|360|
|7|80Byte(Part2) 전송|441|521|200|141|360|
|8|ACK521 수신|521|521|200|141|360|
|9|80Byte(Part3) 전송|521|601|200|141|360|
|10|ACK601 수신|601|601|200|141|360|

> **Server는 140Byte 요청을 수신하면 Receive Buffer에 140Byte가 저장된다. 따라서 Receive Window는 360Byte에서 260Byte(360−140)로 감소한다. 이후 ACK141과 함께 현재 Receive Window(260Byte)를 Client에게 광고한다.**

# 4. Sliding Window 이동 원리

```text
140Byte 전송
      │
      ▼
ACK141 수신
      │
      ▼
SND.UNA 이동
      │
      ▼
Sliding Window 이동
      │
      ▼
다음 데이터 전송
```

Sliding Window는 데이터를 보냈다고 즉시 이동하는 것이 아니라, **ACK를 수신하여 SND.UNA가 앞으로 이동하는 순간 함께 이동한다.**

---

# 5. 왜 280Byte를 한 번에 보내지 않는가?

앞의 예제에서 Client는 Server로부터 **Receive Window = 260Byte**를 광고받았다.

TCP는 수신 측이 광고한 Window 크기를 초과하여 데이터를 전송할 수 없다.

따라서 전송해야 할 데이터가 **280Byte**라면 현재 Window(260Byte)를 초과하므로 한 번에 모두 전송할 수 없다.

예를 들어 다음과 같이 전송된다.

```text
260Byte 전송
      │
      ▼
ACK 수신
      │
      ▼
Window 이동
      │
      ▼
남은 20Byte 전송
```

또는 MSS(Maximum Segment Size) 등의 제약에 따라 여러 개의 TCP Segment로 분할하여 전송될 수도 있다.

중요한 점은 **TCP는 현재 허용된 Sliding Window 범위 내에서만 데이터를 전송하며, ACK를 받아 Window가 이동한 후에 나머지 데이터를 계속 전송한다는 것이다.**

# 6. Wireshark에서 포인터 추론

Wireshark에는 SND.UNA, SND.NXT, RCV.NXT는 직접 표시되지 않는다.

|Wireshark 값|추론 가능한 포인터|
|---|---|
|Sequence Number|SND.NXT|
|ACK Number|SND.UNA, RCV.NXT|
|Window Size|RCV.WND|

예)

```text
Packet1  Seq=1   Ack=241 Win=360
Packet2  Seq=241 Ack=141 Win=200
Packet3  Seq=141 Ack=321 Win=260
Packet4  Seq=321 Ack=141 Win=200
```

Sequence Number, ACK Number, Window Size를 이용하여 각 포인터의 위치를 유추할 수 있다.

---

# 시험 포인트

- ACK Number는 다음에 받을 Byte 번호이다.
- SND.UNA는 ACK를 수신할 때 이동한다.
- SND.NXT는 데이터를 전송할 때 증가한다.
- Window보다 많은 데이터를 전송할 수 없다.
- Sliding Window는 ACK를 기준으로 이동한다.
- Window 크기는 상대방이 광고한 Receive Window에 따라 변경된다.

---

# 핵심 요약

- TCP는 ACK를 이용하여 Sliding Window를 이동시킨다.
- SND.UNA는 가장 오래된 미승인 데이터를 가리킨다.
- SND.NXT는 다음 전송 위치를 의미한다.
- SND.WND는 상대방이 ACK와 함께 광고한 Receive Window를 수신할 때 갱신된다.
- RCV.WND는 Receive Buffer의 남은 공간을 의미한다.
- Window 크기 내에서만 데이터를 전송할 수 있다.
- Wireshark에서는 Sequence Number, ACK Number, Window Size를 이용하여 포인터를 추론할 수 있다.
