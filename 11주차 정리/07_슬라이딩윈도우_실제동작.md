# 학습 목표

이번 장에서는 교수님 PDF의 슬라이딩 윈도우 예제를 통해 TCP가 실제로 데이터를 송수신하면서 SND/RCV 포인터와 Sliding Window가 어떻게 이동하는지 학습한다.

- Sliding Window 이동 원리
- Wireshark를 이용한 포인터 추론

# 1. 초기 상태

TCP 3-Way Handshake가 완료된 직후의 상태이다.

|항목|Client|Server|
|---|---:|---:|
|ISN|1|241|
|Receive Window|360Byte|200Byte|

## Client

```text
SND.UNA = 1
SND.NXT = 1
SND.WND = 360

RCV.NXT = 241
RCV.WND = 200
```

## Server

```text
SND.UNA = 241
SND.NXT = 241
SND.WND = 200

RCV.NXT = 1
RCV.WND = 360
```

> 아직 데이터를 보내지 않았으므로 SND.UNA와 SND.NXT는 같은 값을 가진다.

---

# 2. 전체 동작 흐름 (10단계)

1. 연결 수립
2. Client → 140Byte 요청 전송
3. Server → ACK141 + 80Byte 응답
4. Client → ACK321
5. Server → 파일 Part1(120Byte)
6. Client → ACK441
7. Server → 파일 Part2(80Byte)
8. Client → ACK521
9. Server → 파일 Part3(80Byte)
10. Client → ACK601

# 4. Sliding Window 이동 원리

```text
140Byte 전송
      │
      ▼
ACK141 수신
      │
      ▼
SND.UNA 이동
      │
      ▼
Sliding Window 이동
      │
      ▼
다음 데이터 전송
```

---

# 5. 왜 280Byte를 한 번에 보내지 않는가?

현재 Window가 120Byte뿐이므로 280Byte는

```text
120Byte
↓
80Byte
↓
80Byte
```

처럼 나누어 전송한다.

# 6. Wireshark에서 포인터 추론

Wireshark에는 SND.UNA, SND.NXT, RCV.NXT는 직접 표시되지 않는다.

|Wireshark 값|추론 가능한 포인터|
|---|---|
|Sequence Number|SND.NXT|
|ACK Number|SND.UNA, RCV.NXT|
|Window Size|RCV.WND|

예)

```text
Packet1  Seq=1   Ack=241 Win=360
Packet2  Seq=241 Ack=141 Win=200
Packet3  Seq=141 Ack=321 Win=260
Packet4  Seq=321 Ack=141 Win=200
```

Sequence Number, ACK Number, Window Size를 이용하여 포인터를 유추한다.

---

# 시험 포인트

- ACK Number는 다음에 받을 Byte 번호이다.
- SND.UNA는 ACK 수신 시 이동한다.
- SND.NXT는 전송 시 증가한다.
- Window보다 많이 전송할 수 없다.
- Sliding Window는 ACK를 기준으로 이동한다.

---

# 핵심 요약

- TCP는 ACK를 이용하여 Sliding Window를 이동시킨다.
- SND.UNA는 가장 오래된 미승인 데이터를 가리킨다.
- SND.NXT는 다음 전송 위치를 의미한다.
- Window 크기 내에서만 데이터를 전송한다.
- Wireshark에서는 Sequence Number, ACK Number, Window Size를 이용하여 포인터를 추론한다.
