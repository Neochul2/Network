# Chapter 06 — Sliding Window 포인터 (SND / RCV)

## 학습 목표

- 포인터가 필요한 이유
- SND.UNA / SND.NXT / SND.WND
- RCV.NXT / RCV.WND
- 포인터 간의 관계
- ACK와 포인터의 관계
- Wireshark에서 포인터 추론

---

## 1. 왜 포인터가 필요한가?

TCP는 **Byte Stream 기반 프로토콜**이다.

데이터를 계속 주고받으면서 다음 정보를 항상 추적해야 한다.

| 질문 | 담당 포인터 |
|:---|:---:|
| 어디까지 ACK를 받았는가? | SND.UNA |
| 다음에 무엇을 전송해야 하는가? | SND.NXT |
| 얼마나 더 보낼 수 있는가? | SND.WND |
| 다음에 무엇을 받아야 하는가? | RCV.NXT |
| 얼마나 더 받을 수 있는가? | RCV.WND |

---

## 2. 포인터 한눈에 보기

```
송신(Sender)                    수신(Receiver)
─────────────                   ─────────────
SND.UNA                         RCV.NXT
SND.NXT                         RCV.WND
SND.WND
```

---

## 3. SND.UNA — Sender Unacknowledged

> 아직 ACK를 받지 못한 **가장 오래된 Byte의 Sequence Number**

```
1~100 전송, ACK 미도착

SND.UNA = 1  ← 아직 여기서 대기 중

ACK = 101 도착

SND.UNA = 101  ← 앞으로 이동
```

**택배 비유:** 여러 택배를 보냈는데 가장 오래된 미배달 택배 번호가 SND.UNA다.

---

## 4. SND.NXT — Sender Next

> 다음에 전송할 Byte의 Sequence Number

```
1~100 전송 완료
→ SND.NXT = 101
```

---

## 5. SND.UNA vs SND.NXT

```
[카테고리 1] [카테고리 2] [카테고리 3] [카테고리 4]
              ↑                ↑
           SND.UNA          SND.NXT
```

두 포인터 사이(카테고리 2)에는 **전송했지만 ACK를 받지 못한 데이터**가 있다.

| 포인터 | 가리키는 위치 |
|:---:|:---|
| SND.UNA | 카테고리 2의 첫 번째 Byte |
| SND.NXT | 카테고리 3의 첫 번째 Byte |

---

## 6. SND.WND — Sender Window

> 송신 가능한 Window의 크기(Byte)

수신자가 광고(Advertise)한 Receive Window 값이 그대로 SND.WND가 된다.

```
SND.WND = 500 Byte
→ ACK 없이 최대 500 Byte 범위까지 추가 전송 가능
```

**송신 Window 끝 위치 계산:**

```
송신 Window 끝 = SND.UNA + SND.WND
```

---

## 7. RCV.NXT — Receiver Next

> 수신자가 다음에 받고 싶은 Byte의 번호

```
1~100 수신 완료
→ RCV.NXT = 101
```

ACK Number는 상대방에게 자신의 **RCV.NXT 값을 알려주는 것**이다.

```
ACK = 101
→ "101번부터 보내주세요"
→ RCV.NXT = 101 과 동일한 의미
```

---

## 8. RCV.WND — Receiver Window

> 수신 Buffer에 남아 있는 여유 공간(Byte)

```
Receive Buffer : 1000 Byte
사용 중        :  600 Byte
→ RCV.WND     =  400 Byte
```

송신자는 이 값을 참고해서 전송 가능한 데이터 양을 결정한다.

---

## 9. 포인터 전체 관계도

```
[──카테고리1──][──카테고리2──][──카테고리3──][──카테고리4──]
               ↑              ↑              ↑
            SND.UNA        SND.NXT    SND.UNA + SND.WND
                                      (Window 끝)
```

---

## 10. ACK가 오면 어떻게 되는가?

```
ACK = 101 수신

SND.UNA : 1 → 101  (Window 왼쪽 끝 이동)
SND.NXT : 변화 없음 (아직 전송 중)
Window  : 앞으로 이동 (Sliding!)
```

이것이 **Sliding Window의 핵심 동작**이다.

---

## 11. Wireshark에서 포인터 추론

Wireshark에는 SND.UNA, SND.NXT 같은 이름이 직접 표시되지 않는다.  
대신 아래 값으로 유추할 수 있다.

| Wireshark 필드 | 추론 가능한 포인터 |
|:---|:---:|
| Sequence Number | SND.NXT |
| ACK Number | SND.UNA, RCV.NXT |
| Window Size | RCV.WND |

---

## 시험 포인트

| 포인터 | 의미 |
|:---:|:---|
| SND.UNA | 가장 오래된 미승인 Byte |
| SND.NXT | 다음 전송 Byte |
| SND.WND | 송신 Window 크기 |
| RCV.NXT | 다음 수신 Byte (= ACK Number) |
| RCV.WND | 남은 Receive Buffer 크기 |

- 송신 Window 끝 = **SND.UNA + SND.WND**
- ACK 수신 → **SND.UNA 이동**
- 데이터 전송 → **SND.NXT 증가**
- ACK Number = **RCV.NXT**

---

## 핵심 요약

- TCP는 포인터 5개로 Byte Stream 흐름을 관리한다.
- SND.UNA와 SND.NXT 사이 = 전송했지만 미승인 데이터 (카테고리 2)
- ACK가 도착할 때마다 SND.UNA가 앞으로 이동하며 Window가 슬라이딩된다.
- RCV.NXT는 ACK Number와 동일한 값이다.
- RCV.WND는 수신자가 TCP Header의 Window 필드로 상대방에게 광고한다.
