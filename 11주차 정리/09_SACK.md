# SACK (Selective Acknowledgement)

------------------------------------------------------------------------

# 학습 목표

이번 장에서는 TCP의 **선택적 승인(Selective Acknowledgement, SACK)** 을 학습한다.

학습 후에는 다음 내용을 설명할 수 있어야 한다.

- SACK란 무엇인가?
- 왜 SACK가 필요한가?
- 누적 ACK와의 차이
- SACK의 동작 과정
- TCP Option에서의 SACK
- SACK의 장점

------------------------------------------------------------------------

# 1. SACK란?

SACK(Selective Acknowledgement)은 TCP에서 **수신자가 이미 정상적으로 받은 데이터 구간을 송신자에게 알려주는 기능**이다.

기본 TCP는 **누적 ACK(Cumulative ACK)** 만 사용하지만, SACK는 누적 ACK의 한계를 보완하여 **손실된 Segment만 재전송**할 수 있도록 한다.

SACK는 **TCP Header의 Options 필드**를 사용하며, RFC 2018에서 정의되었다.

> Selective = 선택적  
> Acknowledgement = 승인(확인 응답)

즉,

> "필요한 데이터만 다시 보내세요."

라는 의미이다.

------------------------------------------------------------------------

# 2. 왜 SACK가 필요한가?

TCP는 신뢰성 있는 통신을 제공하기 위해 ACK를 사용한다.

하지만 기본 TCP는 **누적 ACK(Cumulative ACK)** 만 사용한다.

예를 들어 다음과 같이 데이터를 전송했다고 가정하자.

```text
1   2   3   4   5
```

3번 Segment만 손실되었다.

```text
1 ✓
2 ✓
3 ✗
4 ✓
5 ✓
```

수신자는 4와 5를 받았지만,
3이 없기 때문에 데이터를 순서대로 전달할 수 없다.

따라서 계속

```text
ACK = 3
```

만 보낸다.

송신자는

- 4를 받았는지
- 5를 받았는지

알 수 없다.

------------------------------------------------------------------------

# 3. 누적 ACK의 문제점

기본 TCP에서는

```text
ACK = 3
```

만 반복해서 수신한다.

송신자는

```text
3
4
5
```

를 모두 다시 보낼 수도 있다.

이미 정상적으로 받은

```text
4
5
```

도 다시 전송되므로

- 대역폭 낭비
- 네트워크 혼잡 증가
- 처리량 감소

문제가 발생한다.

------------------------------------------------------------------------

# 4. SACK의 등장

SACK를 지원하는 TCP에서는 수신자가 다음과 같이 알려줄 수 있다.

```text
ACK = 3

SACK :
4~5 수신 완료
```

의미는

> "3만 안 왔어요.
> 4와 5는 이미 잘 받았습니다."

송신자는

```text
3
```

만 다시 전송하면 된다.

------------------------------------------------------------------------

# 5. SACK 동작 과정

## Step 1

송신자가 데이터를 전송한다.

```text
1 2 3 4 5
```

------------------------------------------------------------------------

## Step 2

3번 Segment가 손실된다.

```text
1 ✓
2 ✓
3 ✗
4 ✓
5 ✓
```

------------------------------------------------------------------------

## Step 3

수신자는

```text
ACK = 3
```

을 보내면서

```text
SACK

4~5 수신 완료
```

정보도 함께 전송한다.

------------------------------------------------------------------------

## Step 4

송신자는

```text
3
```

만 재전송한다.

이미 받은

```text
4
5
```

는 다시 보내지 않는다.

------------------------------------------------------------------------

# 6. 누적 ACK와 SACK 비교

## 기본 TCP

```text
3 손실

↓

ACK = 3

↓

3 4 5 재전송 가능
```

------------------------------------------------------------------------

## SACK

```text
3 손실

↓

ACK = 3
SACK = 4~5

↓

3만 재전송
```

------------------------------------------------------------------------

# 7. Duplicate ACK와 SACK

TCP에서는 같은 ACK 번호가 여러 번 도착하면 이를 **Duplicate ACK(중복 ACK)** 라고 한다.

예를 들어

```text
ACK = 3
ACK = 3
ACK = 3
```

이 반복되면

송신자는

> "3번 Segment가 손실되었을 가능성이 높다."

고 판단한다.

이때 **Fast Retransmit**이 동작하여 타임아웃을 기다리지 않고 재전송을 시작할 수 있다.

SACK가 함께 사용되면

```text
ACK = 3

SACK = 4~5
```

도 함께 전달되므로

송신자는

> "3만 보내면 된다."

는 사실까지 알 수 있다.

------------------------------------------------------------------------

# 8. SACK는 어디에 저장되는가?

SACK 정보는

**TCP Header의 Options 필드**

에 저장된다.

기본 TCP Header(20Byte)에는 존재하지 않는다.

TCP Header는 다음과 같다.


```text
  0                   15 16                  31
 +----------------------+----------------------+
 | Source Port          | Destination Port     |
 +----------------------+----------------------+
 |               Sequence Number               |
 +---------------------------------------------+
 |          Acknowledgement Number             |
 +----+----------+-----------+-----------------+
 |DOFF| Reserved |   Flags   |     Window      |
 +----------------------+----------------------+
 |     Checksum         |   Urgent Pointer     |
 +----------------------+----------------------+
 |     Options (0~40Byte) + Padding  SACk!!    |
 +---------------------------------------------+
 |              Payload(Data)                  |
 +---------------------------------------------+
```

------------------------------------------------------------------------

# 9. SACK Permitted와 SACK Block

TCP에서는 SACK 사용 전에 먼저

```text
SACK Permitted
```

옵션을 사용한다.

이는

> "나는 SACK 기능을 사용할 수 있습니다."

라는 의미이다.

TCP 연결(3-Way Handshake) 과정에서 서로 SACK 사용 가능 여부를 협상한다.

실제 데이터 전송 중에는

```text
SACK Block
```

이 사용된다.

예를 들어

```text
SACK Block

4001 ~ 6001
```

이라면

> Sequence Number 4001부터 6000까지는 이미 정상적으로 받았다는 의미이다.

------------------------------------------------------------------------

# 10. Wireshark에서는?

Wireshark에서는 TCP Options에서

```text
SACK Permitted
```

또는

```text
SACK
```

```text
Left Edge

Right Edge
```

정보를 확인할 수 있다.

예를 들어

```text
Left Edge : 4001

Right Edge : 6001
```

이라면

4001~6000까지는 이미 수신 완료된 상태이다.

# 11. 실무 예시

1GB 파일을 다운로드하는 중

```text
1
2
3
4
5
```

중에서

```text
3
```

만 손실되었다고 가정하자.

기본 TCP는

```text
3
4
5
```

를 다시 보낼 수도 있다.

하지만 SACK는

```text
3
```

만 재전송한다.

따라서

- 다운로드 속도 향상
- 대역폭 절약
- 서버 부하 감소
- 네트워크 혼잡 감소

효과가 있다.

# 12. 누적 ACK와 SACK 비교

| 구분 | 누적 ACK | SACK |
|------|-----------|------|
| 승인 방식 | 누적 승인 | 선택적 승인 |
| 손실 확인 | 손실 여부만 확인 | 수신 구간까지 확인 |
| 재전송 | 여러 Segment 재전송 가능 | 손실 Segment만 재전송 |
| 대역폭 | 비효율적 | 효율적 |
| 처리량 | 보통 | 우수 |
| TCP Option 사용 | X | O |

# 교수님 수업 포인트

- 기본 TCP는 누적 ACK를 사용한다.
- SACK는 이미 받은 데이터 구간도 알려준다.
- 손실된 Segment만 재전송하도록 도와준다.
- TCP Options 필드를 사용한다.
- Fast Retransmit와 함께 사용되면 성능이 더욱 향상된다.

------------------------------------------------------------------------

# 시험 포인트

- SACK = Selective Acknowledgement
- RFC 2018에서 정의
- TCP Options 필드 사용
- 누적 ACK의 단점을 보완
- 이미 받은 데이터 구간도 송신자에게 알려준다.
- 손실된 Segment만 재전송 가능
- SACK Permitted는 연결 시 협상
- SACK Block은 실제 수신 구간을 표시

# 핵심 요약

- SACK는 Selective Acknowledgement의 약자이다.
- RFC 2018에서 정의되었다.
- 누적 ACK의 한계를 보완한다.
- 수신자는 이미 받은 데이터 구간을 송신자에게 알려준다.
- 송신자는 손실된 Segment만 재전송한다.
- TCP Header의 Options 필드를 사용한다.
- 대용량 파일 전송이나 손실이 많은 환경에서 성능 향상 효과가 크다.
